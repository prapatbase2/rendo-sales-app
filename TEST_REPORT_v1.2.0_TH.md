# รายงานตรวจสอบ Rendo PWA v1.2.0

วันที่ตรวจสอบ: 17 กรกฎาคม 2569

## สาเหตุหลักที่พบ

หน้าลงวันทำงานเดิมใช้ตัวแปรลำดับการโหลดชุดเดียวสำหรับ 2 งานที่ทำพร้อมกัน ได้แก่ การโหลดข้อมูลวันที่เข้าแบบฟอร์ม และการโหลดรายการทั้งเดือน เมื่อผู้ใช้เปลี่ยนวันที่หรือพนักงานเร็ว งานหนึ่งทำให้ผลลัพธ์ของอีกงานถูกยกเลิก แต่ส่วนรายการยังคงแสดง “กำลังโหลด” จึงดูเหมือนแอปค้างและต้องปิดเปิดใหม่

## การตรวจสอบโค้ด

- `node --check app.js` — ผ่าน
- `node --check sw.js` — ผ่าน
- ตรวจ JSON: `manifest.webmanifest`, `version.json`, `firebase.json`, `firestore.indexes.json` — ผ่าน
- ตรวจว่าไม่มีการอ้างตัวแปร `attendanceLoadSeq` เดิม — ผ่าน
- ตรวจ Service Worker, HTML และ asset query string เป็น v1.2.0 — ผ่าน
- ตรวจ Firestore indexes เหลือเฉพาะรายการที่ใช้งานจริง (`salaryAdvances`, `auditLogs`) — ผ่าน

## Browser simulation ด้วย Chromium/Playwright

ใช้หน้าจอมือถือ 390×844 และ Firebase จำลองที่ตั้งใจหน่วง attendance เพื่อสร้างสภาวะชนกันของคำสั่ง

1. Login เจ้าของ — ผ่าน
2. เปิดหน้าลงทำงาน และสั่งเปลี่ยนวันที่ขณะรายการเดือนยังโหลด — ผ่าน; ไม่มี spinner ค้าง
3. สลับหน้า `ยอดขาย → รายเดือน → ลงทำงาน → ยอดขาย` อย่างรวดเร็ว — ผ่าน; หน้าสุดท้ายไม่ถูกผลลัพธ์เก่าเขียนทับ
4. เงินเปิดกะดึงจากยอดปิดล่าสุด 500 บาท — ผ่าน
5. ช่องยอดขายค่า 0 ล้างเมื่อ focus — ผ่าน
6. ช่องว่างคืนเป็น 0 เมื่อ blur — ผ่าน
7. พิมพ์ 250 ได้ทันทีโดยไม่ต้องลบเลข 0 — ผ่าน
8. บันทึกวันทำงาน — ผ่าน; ปุ่มกลับมาใช้งานและรายการเดือนรีเฟรชโดยไม่ค้าง
9. บันทึกวันหยุดร้านในหน้ายอดขาย — ผ่าน; ปุ่มกลับมาใช้งานและสถานะเปลี่ยนเป็นข้อมูลจริง
10. เปิดทุกหน้าของเจ้าของ ได้แก่ Dashboard, ยอดขาย, ลงทำงาน, รายเดือน, เบิกล่วงหน้า, ค่าตอบแทน, ลงรายจ่าย, ประวัติ, ประวัติระบบ, สำรอง, ผู้ใช้ และตั้งค่า — ผ่าน; ไม่พบ JavaScript page error ในชุดทดสอบ

## ข้อจำกัดของการตรวจสอบ

การทดสอบข้างต้นใช้ Browser และ Firebase จำลอง จึงตรวจตรรกะหน้าจอ งานโหลดซ้อน การเปลี่ยนหน้า และการคืนสถานะปุ่มได้ แต่ไม่มีสิทธิ์เข้าถึง Firebase, Google Drive, LINE และมือถือจริงของร้าน จึงต้องทดสอบหลังอัปโหลดด้วยบัญชีจริงอย่างน้อยดังนี้:

1. ลงวันทำงานพร้อมกันจาก 2–3 เครื่อง
2. ตรวจยอดเปิดกะหลังวันหยุดร้านจริง
3. ทดสอบบน Android Chrome และ iPhone Safari
4. ดาวน์โหลด/แชร์ PDF เข้า LINE จากมือถือจริง
5. ตรวจ Auto Backup ในโฟลเดอร์ `Rendo_Backups`

ระบบที่พึ่งเครือข่ายไม่สามารถรับประกันว่าไม่มีข้อผิดพลาดได้ทุกกรณี แต่เวอร์ชันนี้เปลี่ยนจากการรอไม่สิ้นสุดเป็น timeout + ข้อความผิดพลาด + ปุ่มลองใหม่ และป้องกันผลลัพธ์เก่าจากหน้าก่อนหน้า จึงไม่ควรต้องปิดแอปเพียงเพราะคำสั่งหนึ่งตอบช้า
